# Домашнє завдання: Патерн "Репозиторій" та Рефлексія в C#

## Мета завдання

Навчитися реалізовувати архітектурний патерн "Репозиторій" для різних джерел даних (ADO.NET та файлова система), використовувати рефлексію для автоматизації рутинних операцій та забезпечувати слабку пов'язаність компонентів через інтерфейси.

---

## Теоретична частина (Дослідження)

Перед початком написання коду, дослідіть існуючу архітектуру проекту та дайте відповіді на наступні питання (письмово або усно):

1. Яку проблему вирішує використання рефлексії в класі `GenericAdoRepository<T>`?
2. Чому важливо використовувати інтерфейси (наприклад, `IUserRepository`) замість прямих посилань на класи репозиторіїв у бізнес-логіці?
3. Чим відрізняється робота `GenericAdoRepository` від `GenericFileRepository` у поточному проекті?

---

## Практична частина

### 1. Реалізація ADO.NET репозиторіїв

На основі існуючого `UserAdoRepository` та базового класу `GenericAdoRepository<T>`, реалізуйте наступні класи в папці `Persistence/Repositories/AdoImpl/`:

- `PostAdoRepository` (реалізує `IPostRepository`)
- `TagAdoRepository` (реалізує `ITagRepository`)
- `CommentAdoRepository` (реалізує `ICommentRepository`)

**Вимоги:**

- Використовуйте переваги рефлексії базового класу.
- Для специфічних методів (наприклад, `GetBySlug` у `IPostRepository`) пишіть власні SQL-запити, використовуючи метод `Map()` базового класу.

### 2. Завершення File-based репозиторіїв

Дослідіть папку `Persistence/Repositories/FileImpl/`. Переконайтеся, що всі репозиторії коректно зберігають та зчитують дані.

- Реалізуйте відсутню логіку в методах, де зараз стоїть `NotImplementedException`.

### 3. Підтримка декількох СУБД (Multi-DBMS Support)

Проект має підтримувати перемикання між різними базами даних.

- Дослідіть файл `App.config` (або `Program.cs`) та налаштуйте проект так, щоб він міг працювати як мінімум з **двома СУБД** на вибір (наприклад, **SQLite** та **PostgreSQL** або **SQL Server**).
- Переконайтеся, що `GenericAdoRepository<T>` використовує універсальні класи `DbConnection`, `DbCommand` тощо, що дозволяє змінювати провайдера бази даних без переписування логіки репозиторію.
- Опишіть у звіті, чи виникли проблеми з SQL-синтаксисом при переході від однієї СУБД до іншої (наприклад, у назвах таблиць або форматах дат).

### 4. Використання в `Program.cs`

Продемонструйте роботу створених репозиторіїв у головному файлі програми:

- Створіть екземпляри репозиторіїв через їх інтерфейси.
- Виконайте повний цикл CRUD для однієї з нових сутностей (наприклад, додайте новий Пост, отримайте його за ID, оновіть заголовок та видаліть).
- Спробуйте змінити реалізацію з ADO на File (наприклад, змінити `IUserRepository repo = new UserAdoRepository(...)` на `IUserRepository repo = new UserFileRepository(...)`) і переконайтеся, що інша частина коду не потребує змін.

## Критерії оцінювання

1. **Коректність успадкування**: Репозиторії мають наслідувати правильні базові класи та інтерфейси.
2. **Чистота коду**: Відсутність дублювання логіки, яка вже є в `GenericAdoRepository`.
3. **Робота з SQL**: Правильне використання параметрів (`SqlParameter`) для запобігання SQL-ін'єкціям.
4. **Типізація**: Правильне використання `Guid` та `DateTime`.
